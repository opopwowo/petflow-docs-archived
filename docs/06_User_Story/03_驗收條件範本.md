# 驗收條件（Acceptance Criteria）範本

> 提供 Given／When／Then 標準範本、撰寫指引與常見情境模板（權限不足、跨租戶存取、軟刪除、稽核紀錄），確保全專案驗收條件風格一致、可測試、可自動化。

## 1. 基本格式（Gherkin 風格）

### 1.1 標準三段式

```gherkin
Given（前置條件）：系統與資料的初始狀態、操作者身分與權限
When（觸發動作）：使用者或系統執行的單一動作
Then（預期結果）：可客觀驗證的結果（狀態、回應碼、畫面、副作用）
```

範例（取自 [US-PET-001](02_各模組使用者故事清單.md#us-pet-001-建立寵物檔案)）：

> Given 小美已登入且具 `pet:create` 權限，When 她輸入名稱與物種並送出，Then 系統建立寵物檔案、回傳 201，且列表可見該寵物。

### 1.2 進階句型

| 句型 | 用途 | 範例 |
| --- | --- | --- |
| `And` | 補充同段落的多個條件／結果 | Then 回傳 201，And Audit Log 新增一筆建立紀錄 |
| `But` | 強調不應發生的事 | Then 顯示錯誤訊息，But 不寫入任何資料 |
| Scenario Outline | 同一行為、多組資料 | 以表格列舉物種／孕期天數等參數組合 |

### 1.3 撰寫規則

1. **每則故事至少 2 條**：至少 1 條正向（Happy Path）＋建議 1 條例外／邊界。
2. **一條一情境**：一組 Given/When/Then 只驗證一件事；避免在單條塞入多個 When。
3. **可觀察、可量測**：以回應碼（200/201/204/400/401/403/404/409/422）、狀態值、畫面元素、資料變化描述結果；禁止「很快」「友善」「正確」等不可驗證形容詞。
4. **描述行為不描述實作**：寫「回傳 409 並提示既有寵物」，不寫「D1 加 unique index」。
5. **使用統一語言**：實體名稱依 Ubiquitous Language（Pet、Owner、Vaccination、MedicalRecord、BreedingRecord、RegistrationApplication、Tenant、Store）。
6. **對應測試**：每條驗收條件須可轉為 [30 測試](../30_測試/README.md) 之自動化測試案例。

### 1.4 常見反模式

| 反模式 | 錯誤範例 | 修正 |
| --- | --- | --- |
| 不可測 | Then 使用者覺得方便 | Then 建檔流程於單一頁面完成且必填欄位僅 2 個 |
| 綁定實作 | When 系統寫入 pets 資料表 | When 小美送出建檔表單 |
| 多重觸發 | When 建立又編輯又刪除 | 拆為三條情境 |
| 缺前置 | 直接寫 When/Then | 補 Given：身分、權限、資料初始狀態 |
| 只有 Happy Path | 僅一條成功情境 | 補權限不足、資料衝突、驗證失敗等例外 |

## 2. 常見情境模板

以下四類為 PetFlow Enterprise 之橫切關注點（見 [01 規範第 5 節](01_User_Story撰寫規範_INVEST.md#5-橫切關注點每則故事隱含之全域規則)）。撰寫故事時**視風險擇要明寫**；未明寫者於驗收與測試時仍一律適用。使用時將 `<占位符>` 代入實際角色、實體與動作。

### 2.1 模板 A：權限不足（RBAC）

```gherkin
Given <角色> 已登入但不具 <permission>（如 `pet:delete`）權限
When <角色> 透過介面或直接呼叫 API 執行 <動作>
Then 系統回傳 403、不執行任何業務邏輯
And 回應採統一錯誤格式，不洩漏資源存在性與內部細節
```

範例：Given 小美（STAFF）不具 `pet:delete` 權限，When 她直接呼叫刪除寵物 API，Then 回傳 403 且寵物資料不受影響。

補充變體：

- **介面層**：Given 使用者無 <permission>，When 開啟頁面，Then 對應按鈕／入口不顯示（隱藏而非僅 disabled）。
- **Deny by default**：Given 端點未宣告所需權限，When 任何請求到達，Then 一律拒絕。

### 2.2 模板 B：跨租戶存取（Multi-Tenant）

```gherkin
Given <角色> 之 Token 綁定租戶 A 上下文
When <角色> 以租戶 B 之 <實體> ID 請求讀取／寫入
Then 系統回傳 404（或 403），不回傳、不變更租戶 B 任何資料
And 回應不可讓呼叫端推斷該 ID 是否存在
```

範例：Given 雅婷屬於租戶 A，When 她以租戶 B 的 Pet ID 請求詳情，Then 回傳 404。

補充變體：

- **列表查詢**：Given 兩租戶皆有資料，When 查詢任何列表，Then 結果僅含本租戶資料（強制帶 `tenantId`）。
- **多租戶切換**（如 Dr. Chen）：Given 使用者切換至租戶 B，Then 原租戶 A 之資源即刻不可存取。

### 2.3 模板 C：軟刪除（Soft Delete）

```gherkin
（刪除）
Given <實體> 存在且 <角色> 具 <entity>:delete 權限
When <角色> 刪除該 <實體>
Then <實體> 被標記 deleted_at、所有預設查詢（列表／搜尋／關聯顯示）不再包含它
And 可於回收區檢視

（還原）
Given <實體> 位於回收區
When 具權限之 <角色> 執行還原
Then <實體> 恢復於原查詢範圍，關聯與衍生計算（如容量）一併回復

（防護）
Given <實體> 已被軟刪除
When 任何人以其 ID 直接請求
Then 預設回傳 404（除回收區／稽核情境外）
```

範例：見 [US-PET-006](02_各模組使用者故事清單.md#us-pet-006-軟刪除與還原寵物)、[US-PHT-005](02_各模組使用者故事清單.md#us-pht-005-軟刪除與還原照片)。

> 硬刪除須走特別流程並記錄 Audit Log，不得作為一般故事之驗收條件。

### 2.4 模板 D：稽核紀錄（Audit Log）

```gherkin
Given <角色> 完成一筆 <建立｜修改｜刪除｜還原> 操作
When 具 `audit:read` 權限者查詢稽核紀錄
Then 可見一筆對應紀錄，內容包含：
  who（操作者）、what（動作與實體）、when（時間）、
  where（IP／裝置）、before/after（變更內容）、tenantId
And 該紀錄無法被任何角色更新或刪除（append-only）
```

範例：Given 小美修改寵物體重，When 雅婷查詢稽核，Then 可見 before/after 之體重值對照。

補充變體：

- **失敗操作**：Given 寫入因 422/409 未生效，Then 不產生資料變更紀錄（或標記為失敗嘗試）。
- **敏感讀取**：Given 使用者讀取完整個資（`owner:pii:read`），Then 讀取行為留有稽核紀錄。

## 3. 輸入驗證與錯誤回應模板

```gherkin
（必填缺漏）
Given <角色> 未填寫必填欄位 <欄位>
When 送出表單／請求
Then 回傳 422 並逐欄提示缺漏，不寫入任何資料

（唯一性衝突）
Given 租戶內已存在相同 <唯一欄位>（如晶片號碼）之 <實體>
When 以相同值建立
Then 回傳 409 並提示既有資料連結

（狀態機非法轉移）
Given <實體> 目前狀態為 <狀態A>
When 請求轉移至不允許之 <狀態X>
Then 回傳 422 並列出允許之下一步狀態
```

## 4. 驗收條件自檢清單

提交故事前逐項確認：

- [ ] 至少 2 條，含 1 條正向與 1 條例外／邊界（次要故事得為 1–2 條）。
- [ ] 每條皆為單一情境、單一 When。
- [ ] 結果可客觀驗證（回應碼／狀態／資料變化），無模糊形容詞。
- [ ] 未綁定技術實作細節。
- [ ] 涉及刪除者採軟刪除模板；涉及權限、個資、跨租戶風險者明寫對應模板。
- [ ] 名詞符合統一語言與 [05 使用者角色](../05_使用者角色/README.md) 之角色定義。
- [ ] 每條可對應至少一個測試案例。

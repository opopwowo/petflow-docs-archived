## 2. 統一錯誤回應格式

依 `CLAUDE.md` API First 規範，所有 `/api/v1` 端點的錯誤回應採統一結構：

```json
{
  "success": false,
  "error": {
    "code": "PERMISSION_DENIED",
    "excId": "EXC-02",
    "message": "您沒有執行此操作的權限。",
    "details": [
      { "field": "name", "issue": "必填欄位不得為空" }
    ],
    "traceId": "01J8ZK3Q9W..."
  }
}
```

| 欄位 | 說明 |
| --- | --- |
| `error.code` | 機器可讀錯誤碼（`UPPER_SNAKE_CASE`，見第 3 節各例外定義） |
| `error.excId` | 對應本目錄之例外代碼（EXC-NN），供除錯與測試對照 |
| `error.message` | 給使用者的繁體中文訊息，遵循第 4 節訊息原則 |
| `error.details` | 選填；僅 422 驗證失敗時逐欄位列出問題 |
| `error.traceId` | 請求追蹤識別碼，供客服與維運查詢日誌 |

## 3. 例外目錄

### EXC-01 未登入或 Token 失效 — 401

| 項目 | 內容 |
| --- | --- |
| 情境 | 未附帶 Token、Token 過期、簽章無效、Token 缺少租戶上下文 |
| 系統行為 | 拒絕請求；不執行任何商業邏輯；前端導向登入頁並保留原目的地 |
| HTTP | 401 Unauthorized |
| error.code | `UNAUTHENTICATED` |
| 訊息原則 | 「請先登入」；不得說明 Token 失效的具體原因（過期／偽造不區分） |
| 稽核 | 不寫入 Audit Log（未識別操作者）；異常頻率由 WAF／速率限制監控 |

### EXC-02 權限不足 — 403

| 項目 | 內容 |
| --- | --- |
| 情境 | RBAC 拒絕（缺少 `資源:動作` 權限）、操作高於自身層級的角色、VET 存取未授權寵物、租戶遭停用／凍結 |
| 系統行為 | 拒絕請求；記錄拒絕事件；連續大量 403 觸發告警（見 26 通知中心） |
| HTTP | 403 Forbidden |
| error.code | `PERMISSION_DENIED` |
| 訊息原則 | 「您沒有執行此操作的權限」；不得列出取得該權限所需的角色細節 |
| 稽核 | 寫入 Audit Log（`who`、`what`、被拒權限），供安全稽核 |

### EXC-03 跨租戶存取 — 404（隱藏存在性）

| 項目 | 內容 |
| --- | --- |
| 情境 | 請求之資源 ID 存在但屬於其他租戶 |
| 系統行為 | 一律回應與「資源不存在」完全相同的結果，**不洩漏資源存在性**；Repository 層以 `tenantId` 過濾，此情境在資料層即不可見 |
| HTTP | 404 Not Found |
| error.code | `RESOURCE_NOT_FOUND`（與 EXC-04 相同，刻意不可區分） |
| 訊息原則 | 「找不到指定的資料」；回應內容、時間差不得暴露與 EXC-04 的差異 |
| 稽核 | 寫入安全事件日誌（疑似探測行為），高頻觸發告警 |

### EXC-04 資源不存在或已軟刪除 — 404

| 項目 | 內容 |
| --- | --- |
| 情境 | ID 不存在、資源已軟刪除（`deleted_at` 非空）且非還原／回收站情境 |
| 系統行為 | 拒絕請求；還原類 Use Case 與回收站查詢例外地可見軟刪除資料（受 `資源:restore` 權限保護） |
| HTTP | 404 Not Found |
| error.code | `RESOURCE_NOT_FOUND` |
| 訊息原則 | 「找不到指定的資料」；不區分「從未存在」與「已刪除」 |
| 稽核 | 不寫入（一般讀取失誤）；批次異常由監控處理 |

### EXC-05 狀態衝突 — 409

| 項目 | 內容 |
| --- | --- |
| 情境 | 非法狀態轉移（如已歿寵物改為待售，違反 08 流程圖之狀態機）、唯一性衝突（同租戶晶片號碼重複、Email 重複）、還原未被刪除的資源 |
| 系統行為 | 拒絕請求並說明衝突原因；不做部分寫入（交易整體回滾） |
| HTTP | 409 Conflict |
| error.code | `STATE_CONFLICT`（狀態機）／`DUPLICATE_RESOURCE`（唯一性） |
| 訊息原則 | 明確指出衝突欄位或不允許的狀態轉移（如「晶片號碼已被使用」），協助使用者修正 |
| 稽核 | 寫入 Audit Log（嘗試之變更內容） |

### EXC-06 輸入驗證失敗 — 422

| 項目 | 內容 |
| --- | --- |
| 情境 | Schema 驗證不通過（必填缺漏、格式錯誤、超出長度）、關聯資料不齊備（寵物未指定主要飼主、引用的範本不存在） |
| 系統行為 | 進入 Domain 前即攔截（見 CLAUDE.md TypeScript Strict 規範）；`error.details` 逐欄位回報 |
| HTTP | 422 Unprocessable Entity |
| error.code | `VALIDATION_FAILED` |
| 訊息原則 | 逐欄位、可操作（「出生日期不得晚於今日」）；不得回傳原始 schema 錯誤字串 |
| 稽核 | 不寫入（未進入商業邏輯） |

### EXC-07 方案額度超限 — 422

| 項目 | 內容 |
| --- | --- |
| 情境 | 寵物數／使用者數／照片容量／門市數達方案上限、租戶處於到期寬限唯讀模式、方案不含目標功能（如自訂角色為 Enterprise 限定） |
| 系統行為 | 拒絕請求並附升級／續約引導（連結至 19 會員訂閱）；寬限期內既有資料唯讀可存取 |
| HTTP | 422 Unprocessable Entity |
| error.code | `PLAN_LIMIT_EXCEEDED`／`PLAN_FEATURE_UNAVAILABLE`／`SUBSCRIPTION_EXPIRED` |
| 訊息原則 | 說明目前方案、已用額度與上限，並提供升級行動（「目前方案上限 200 隻，升級 Pro 可管理 1,000 隻」） |
| 稽核 | 寫入產品分析事件（升級漏斗訊號，見 02 市場分析 GTM 漏斗） |

### EXC-08 併發修改衝突 — 409

| 項目 | 內容 |
| --- | --- |
| 情境 | 兩位使用者同時編輯同一資源，後送出者的版本標記（`updatedAt`／version）與現況不符 |
| 系統行為 | 採樂觀鎖：拒絕後送出者；回傳最新版本供前端顯示差異，由使用者決定覆寫或放棄 |
| HTTP | 409 Conflict |
| error.code | `CONCURRENT_MODIFICATION` |
| 訊息原則 | 「資料已被其他使用者更新，請重新載入後再試」；標示更新者與時間（同租戶內） |
| 稽核 | 寫入 Audit Log（成功方已記錄；衝突事件另記嘗試者） |

## 4. HTTP 狀態碼使用原則

依 `CLAUDE.md` 第 6 節 API First 規範：

| 狀態碼 | 用途 | 對應例外 |
| --- | --- | --- |
| 200 | 查詢／更新成功 | — |
| 201 | 建立成功 | — |
| 204 | 刪除／還原成功（無回應內容） | — |
| 400 | 請求格式錯誤（無法解析的 JSON、錯誤的參數型別） | 保留；語意驗證一律用 422 |
| 401 | 未驗證 | EXC-01 |
| 403 | 已驗證但無權限 | EXC-02 |
| 404 | 不存在／不可見（含跨租戶） | EXC-03、EXC-04 |
| 409 | 與現有狀態衝突 | EXC-05、EXC-08 |
| 422 | 語意驗證失敗／方案限制 | EXC-06、EXC-07 |
| 500 | 未預期的系統錯誤 | 見第 5 節 |

## 5. 系統錯誤（500）處理原則

- 未預期錯誤一律回 500，`error.code` 為 `INTERNAL_ERROR`，**不得**將堆疊、SQL、內部路徑洩漏給客戶端。
- 每筆 500 皆附 `traceId` 並觸發維運告警（見 [29 部署](../29_部署/README.md) 可觀測性規劃）。
- 寫入類操作發生 500 時交易回滾，不留部分寫入；佇列類任務（通知、影像處理）依 Queues 重試策略處理。

## 6. 訊息撰寫通則

1. 一律繁體中文、以使用者行動為導向（說「怎麼辦」，不只說「錯了」）。
2. 安全優先：401／403／404 不洩漏系統內部資訊與資源存在性；422／409 則盡量具體。
3. 同一例外在 Web／行動裝置顯示一致文案，文案集中管理（i18n 資源檔，為未來多語系預留）。
4. 錯誤訊息不包含個資與敏感資料（晶片號碼等識別資訊僅在使用者本就有權檢視時顯示）。

# 狀態機圖

> 以 Mermaid stateDiagram-v2 定義關鍵實體的狀態機：寵物生命週期（含軟刪除）、官方登記申請、訂閱、付款與配種，作為資料庫欄位與 API 行為的設計依據。

| 文件版本 | 狀態 | 最後更新 | 所屬模組 |
| --- | --- | --- | --- |
| v0.2.0 | 初稿 | 2026-07-02 | 08 流程圖 |

---

## 1. 文件目的

狀態機是商業規則的核心，屬於 **Domain 層**的職責。本文件定義各 Aggregate Root 的合法狀態與轉移條件；**未列於本文件的狀態轉移一律視為非法**，API 收到非法轉移請求時回應 `409 Conflict` 或 `422 Unprocessable Entity`。

**共通規則：**

- 每次狀態轉移均為寫入操作，必須記錄 Audit Log（含 before/after 狀態）。
- 狀態值以 `UPPER_SNAKE_CASE` 常數落入資料庫欄位，命名須與本文件一致（統一語言）。
- 軟刪除（`deleted_at`）為**橫切狀態（cross-cutting）**，不屬於任一實體狀態機的節點：任何狀態的實體都可被軟刪除與還原，還原後回到刪除前的原狀態；已軟刪除資料預設排除於所有查詢。

## 2. 寵物狀態機（Pet，全 repo 標準）

### 2.1 說明

- **觸發點**：寵物建檔即進入「在養 ACTIVE」。
- **關鍵決策**：是否上架待售、是否成交、是否死亡。
- **例外處理**：SOLD 與 DECEASED 為終態；誤操作更正僅限 ADMIN 以上角色走特別流程並記 Audit Log。軟刪除為橫切狀態，於下方獨立描述。

### 2.2 狀態圖

```mermaid
stateDiagram-v2
    [*] --> ACTIVE : 建檔完成（入店 / 出生 / 寄售）
    ACTIVE : 在養 ACTIVE
    FOR_SALE : 待售 FOR_SALE
    SOLD : 已售出 SOLD
    DECEASED : 已歿 DECEASED

    ACTIVE --> FOR_SALE : 上架待售（設定售價）
    FOR_SALE --> ACTIVE : 撤回上架（自留 / 暫停販售）
    FOR_SALE --> SOLD : 成交並完成交付與過戶
    ACTIVE --> DECEASED : 死亡（記錄死因與日期）
    FOR_SALE --> DECEASED : 死亡（自動下架並記錄）
    SOLD --> [*]
    DECEASED --> [*]

    note right of SOLD
        終態。誤售更正僅限 ADMIN 以上
        走「狀態更正」特別流程並記 Audit Log
    end note
    note right of DECEASED
        終態。檔案保留供追溯，
        疫苗提醒等排程一律停止
    end note
```

### 2.3 軟刪除橫切狀態

```mermaid
stateDiagram-v2
    state "任一業務狀態<br/>ACTIVE / FOR_SALE / SOLD / DECEASED" as ANY
    state "已軟刪除<br/>deleted_at 有值" as DELETED

    [*] --> ANY
    ANY --> DELETED : 軟刪除（記錄操作者與時間）
    DELETED --> ANY : 還原 restore（回到刪除前原狀態）
    DELETED --> [*] : 硬刪除（特別流程，需審核並記 Audit Log）
```

**軟刪除規則：**

| 規則 | 內容 |
| --- | --- |
| 標記方式 | `deleted_at` 時間戳記（NULL 表示未刪除） |
| 查詢行為 | 所有查詢預設排除已軟刪除資料 |
| 還原 | 清空 `deleted_at`，回到刪除前的業務狀態 |
| 硬刪除 | 僅限特別流程（如個資刪除請求），須審核並完整記錄 Audit Log |
| 連動 | 寵物被軟刪除時，其疫苗提醒、待辦排程一併暫停 |

## 3. 官方登記申請狀態機（RegistrationApplication）

### 3.1 說明

- **觸發點**：建立登記申請即進入「草稿 DRAFT」。
- **關鍵決策**：文件是否齊全並完成晶片植入確認、主管機關審查結果（核准 / 補件 / 駁回）。
- **例外處理**：補件為替代流程，逾期未補件轉「已撤件」；駁回可申覆（回到審查中）或關閉後重新申請。

### 3.2 狀態圖

```mermaid
stateDiagram-v2
    [*] --> DRAFT : 建立申請
    DRAFT : 草稿 DRAFT
    PREPARING : 準備文件 PREPARING
    CHIP_CONFIRMED : 晶片植入已確認 CHIP_CONFIRMED
    SUBMITTED : 已送件 SUBMITTED
    UNDER_REVIEW : 審查中 UNDER_REVIEW
    SUPPLEMENT_REQUIRED : 待補件 SUPPLEMENT_REQUIRED
    APPROVED : 已核准 APPROVED
    REJECTED : 已駁回 REJECTED
    CERTIFICATE_ISSUED : 已發證 CERTIFICATE_ISSUED
    WITHDRAWN : 已撤件 WITHDRAWN

    DRAFT --> PREPARING : 開始準備文件
    PREPARING --> CHIP_CONFIRMED : 晶片植入並掃描確認
    CHIP_CONFIRMED --> SUBMITTED : 送件至主管機關
    SUBMITTED --> UNDER_REVIEW : 機關收件
    UNDER_REVIEW --> SUPPLEMENT_REQUIRED : 通知補件（替代流程）
    SUPPLEMENT_REQUIRED --> SUBMITTED : 期限內補齊重送
    SUPPLEMENT_REQUIRED --> WITHDRAWN : 逾期未補件（視同撤件）
    UNDER_REVIEW --> APPROVED : 審查核准
    UNDER_REVIEW --> REJECTED : 審查駁回（記錄原因）
    REJECTED --> UNDER_REVIEW : 申覆成立
    APPROVED --> CERTIFICATE_ISSUED : 發證（登錄證號與日期）
    DRAFT --> WITHDRAWN : 申請人撤回
    PREPARING --> WITHDRAWN : 申請人撤回
    CERTIFICATE_ISSUED --> [*]
    REJECTED --> [*] : 放棄申覆（可另建新申請）
    WITHDRAWN --> [*] : 可另建新申請
```

> 補充：每次狀態變更均發送站內與 Email 通知承辦人；`SUPPLEMENT_REQUIRED` 帶補件期限欄位，由排程於期限前提醒。

## 4. 訂閱狀態機（Subscription）

### 4.1 說明

- **觸發點**：租戶開通時建立訂閱（Free 直接生效；付費方案待付款成功）。
- **關鍵決策**：續費付款是否成功、是否排定降級、是否取消。
- **例外處理**：續費失敗進入「寬限期 PAST_DUE」（預設 7 天，期間重試扣款），寬限期滿未付款自動降回 Free。

### 4.2 狀態圖

```mermaid
stateDiagram-v2
    [*] --> TRIALING : 開通（如提供試用）
    [*] --> ACTIVE : 開通（Free 或付款成功）
    TRIALING : 試用中 TRIALING
    ACTIVE : 生效中 ACTIVE
    PENDING_DOWNGRADE : 已排定降級 PENDING_DOWNGRADE
    PAST_DUE : 寬限期 PAST_DUE
    CANCELED : 已取消 CANCELED

    TRIALING --> ACTIVE : 試用期滿轉正（付款成功）
    TRIALING --> CANCELED : 試用期滿未轉正
    ACTIVE --> ACTIVE : 升級（即時生效，按比例補差價）
    ACTIVE --> PENDING_DOWNGRADE : 排定降級（下期生效）
    PENDING_DOWNGRADE --> ACTIVE : 取消降級 / 週期日切換至新方案
    ACTIVE --> PAST_DUE : 續費扣款失敗
    PAST_DUE --> ACTIVE : 寬限期內補款成功
    PAST_DUE --> ACTIVE : 寬限期滿自動降回 Free
    ACTIVE --> CANCELED : 租戶主動取消（期末生效）
    CANCELED --> ACTIVE : 重新訂閱
    CANCELED --> [*]

    note right of PAST_DUE
        寬限期預設 7 天，
        期間每日重試扣款並通知 OWNER
    end note
```

## 5. 付款狀態機（Payment）

### 5.1 說明

- **觸發點**：訂閱開通、升級補差價或週期續費時建立付款單。
- **關鍵決策**：金流閘道回傳結果、是否重試、是否退款。
- **例外處理**：授權逾時視為失敗；退款僅限成功付款且於政策期限內；爭議款（chargeback）凍結並人工處理。

### 5.2 狀態圖

```mermaid
stateDiagram-v2
    [*] --> PENDING : 建立付款單
    PENDING : 待付款 PENDING
    PROCESSING : 處理中 PROCESSING
    SUCCEEDED : 付款成功 SUCCEEDED
    FAILED : 付款失敗 FAILED
    REFUNDING : 退款處理中 REFUNDING
    REFUNDED : 已退款 REFUNDED
    DISPUTED : 爭議中 DISPUTED

    PENDING --> PROCESSING : 送出至金流閘道
    PENDING --> FAILED : 逾時未付款（自動關單）
    PROCESSING --> SUCCEEDED : 閘道回傳成功（Webhook 驗證）
    PROCESSING --> FAILED : 閘道回傳失敗 / 授權逾時
    FAILED --> PENDING : 使用者重試（建立新嘗試）
    SUCCEEDED --> REFUNDING : 申請退款（政策期限內）
    REFUNDING --> REFUNDED : 退款完成
    REFUNDING --> SUCCEEDED : 退款遭拒（維持原狀）
    SUCCEEDED --> DISPUTED : 爭議款 chargeback
    DISPUTED --> REFUNDED : 爭議成立
    DISPUTED --> SUCCEEDED : 爭議不成立
    SUCCEEDED --> [*]
    REFUNDED --> [*]
    FAILED --> [*]
```

> 補充：Webhook 必須驗簽且具冪等性（同一事件重送不得重複轉移狀態），細節見 [20 付款系統](../20_付款系統/README.md)。

## 6. 配種狀態機（BreedingRecord）

### 6.1 說明

- **觸發點**：MANAGER / VET 建立配種計畫，選定種公與種母。
- **關鍵決策**：健康評估是否通過、交配後是否確認懷孕、生產結果。
- **例外處理**：健康評估未通過即取消；未受孕可重新排配或結案；流產/難產轉入醫療流程並記錄結果。

### 6.2 狀態圖

```mermaid
stateDiagram-v2
    [*] --> PLANNED : 建立配種計畫
    PLANNED : 計畫中 PLANNED
    HEALTH_CHECKED : 健康評估通過 HEALTH_CHECKED
    MATED : 已交配 MATED
    PREGNANT : 已確認懷孕 PREGNANT
    NOT_PREGNANT : 未受孕 NOT_PREGNANT
    DELIVERED : 已生產 DELIVERED
    CANCELED : 已取消 CANCELED

    PLANNED --> HEALTH_CHECKED : 種公母健康評估通過（VET）
    PLANNED --> CANCELED : 健康評估未通過 / 計畫取消
    HEALTH_CHECKED --> MATED : 完成交配（記錄日期）
    MATED --> PREGNANT : 懷孕確認（超音波 / 觸診）
    MATED --> NOT_PREGNANT : 確認未受孕
    NOT_PREGNANT --> PLANNED : 重新排配
    NOT_PREGNANT --> [*] : 結案
    PREGNANT --> DELIVERED : 生產（記錄胎數，幼崽建檔為 Pet）
    PREGNANT --> CANCELED : 流產（轉醫療流程並記錄）
    DELIVERED --> [*]
    CANCELED --> [*]

    note right of DELIVERED
        幼崽逐一建立 Pet 檔案並回鏈
        BreedingRecord（父母、胎次）
    end note
```

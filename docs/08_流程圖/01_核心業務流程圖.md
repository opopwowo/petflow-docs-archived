# 核心業務流程圖

> 以 Mermaid 流程圖完整描述 PetFlow Enterprise 的核心業務流程：租戶開通、寵物入店建檔、健康管理、官方登記、售出/交付、疫苗提醒與訂閱方案變更。

## 1. 文件目的

本文件為 PetFlow Enterprise 全平台核心業務流程的唯一事實來源（SSOT），供產品、設計、開發與測試團隊對齊流程認知。所有流程圖統一使用 Mermaid 語法，命名與繪製規範見 [04_流程圖繪製與命名規範](04_流程圖繪製與命名規範.md)。

**共通前提（適用本文件所有流程）：**

- 所有操作均在 **Multi-Tenant 隔離**下進行，任何查詢與寫入一律帶 `tenantId`。
- 所有操作均受 **RBAC** 權限檢查（角色：SUPER_ADMIN、OWNER、ADMIN、MANAGER、STAFF、VET、VIEWER），預設 Deny by default。
- 所有寫入操作（建立/修改/刪除/還原）一律記錄 **Audit Log**。
- 流程中的「刪除」一律為**軟刪除**（`deleted_at`），並提供還原能力。
- 通知管道：站內通知、Email、推播（LINE 規劃中）。

## 2. 租戶開通（Onboarding）流程

### 2.1 流程說明

- **觸發點**：潛在客戶（寵物店、連鎖門市、專業繁殖者、寵物服務業者）於官網註冊，或由業務代為建立。
- **關鍵決策**：Email 驗證是否通過、選擇訂閱方案（Free / Starter / Pro / Enterprise）、付費方案是否付款成功。
- **例外處理**：Email 驗證逾時可重發驗證信；付款失敗可重試或先降回 Free 方案開通；Enterprise 方案需業務審核後開通。

### 2.2 流程圖

```mermaid
flowchart TD
    A(["開始：客戶註冊"]) --> B["填寫租戶資料<br/>店名 / 統編 / 聯絡人 / Email"]
    B --> C["寄送 Email 驗證信"]
    C --> D{"Email 驗證通過？"}
    D -- "逾時 / 失敗" --> C2["重發驗證信<br/>（最多 3 次）"]
    C2 --> D
    D -- "通過" --> E["選擇訂閱方案"]
    E --> F{"方案類型？"}
    F -- "Free NT$0" --> H["建立 Tenant 與預設 Store"]
    F -- "Starter / Pro" --> G["線上付款"]
    F -- "Enterprise NT$3,999 起" --> G2["業務審核與合約簽署"]
    G --> G3{"付款成功？"}
    G3 -- "失敗" --> G4{"重試付款？"}
    G4 -- "是" --> G
    G4 -- "否" --> G5["以 Free 方案先行開通<br/>保留升級入口"]
    G5 --> H
    G3 -- "成功" --> H
    G2 --> G6{"審核通過？"}
    G6 -- "否" --> Z1(["結束：婉拒並通知業務跟進"])
    G6 -- "是" --> H
    H --> I["建立 OWNER 帳號<br/>套用預設 RBAC 角色"]
    I --> J["初始化租戶設定<br/>品種字典 / 通知偏好 / 方案配額"]
    J --> K["發送歡迎通知<br/>站內 + Email"]
    K --> L["引導精靈：建立門市、邀請成員、匯入寵物資料"]
    L --> M(["結束：租戶啟用完成"])
```

> 補充：Onboarding 全程寫入 Audit Log（`who=系統/業務`、`what=TenantCreated` 等事件）。租戶停用不刪除資料，僅標記狀態並凍結登入。

## 3. 核心業務主流程：寵物入店建檔 → 健康 → 登記 → 售出/交付

### 3.1 流程說明

- **觸發點**：寵物入店（自行繁殖出生、進貨、飼主寄養轉售）。
- **關鍵決策**：來源類型、是否植入晶片、是否辦理官方登記、是否上架待售。
- **例外處理**：健康檢查異常轉入醫療流程（MedicalRecord）；寵物不幸死亡轉入「已歿」結案流程；建檔資料錯誤以修改方式修正並記 Audit Log，不做實體刪除。
- **狀態對應**：本流程驅動寵物狀態機（在養 ACTIVE → 待售 FOR_SALE → 已售出 SOLD / 已歿 DECEASED），狀態機詳見 [02_狀態機圖](02_狀態機圖.md)。

### 3.2 流程圖

```mermaid
flowchart TD
    A(["開始：寵物入店"]) --> B{"來源？"}
    B -- "自行繁殖" --> B1["關聯 BreedingRecord<br/>帶入父母與胎次資訊"]
    B -- "外部進貨" --> B2["登錄來源憑證<br/>供應商 / 檢疫證明"]
    B -- "飼主寄售" --> B3["建立 Owner 關聯<br/>簽署寄售協議"]
    B1 --> C["建立 Pet 檔案<br/>品種 / 性別 / 生日 / 毛色 / 照片"]
    B2 --> C
    B3 --> C
    C --> C1["狀態：在養 ACTIVE"]
    C1 --> D["初次健康檢查<br/>由 VET 角色執行"]
    D --> E{"健康狀況？"}
    E -- "異常" --> E1["建立 MedicalRecord<br/>治療 / 隔離 / 追蹤"]
    E1 --> E2{"康復？"}
    E2 -- "是" --> F
    E2 -- "不幸死亡" --> E3["狀態：已歿 DECEASED<br/>記錄死因與日期"]
    E3 --> Z1(["結束：結案歸檔"])
    E -- "正常" --> F["建立疫苗排程 Vaccination<br/>啟動疫苗提醒流程"]
    F --> G{"是否植入晶片？"}
    G -- "是" --> G1["登錄晶片號碼"]
    G -- "否" --> H
    G1 --> H{"是否辦理官方登記？"}
    H -- "是" --> H1["啟動官方登記申請流程<br/>詳見第 4 章"]
    H -- "否" --> I
    H1 --> I{"是否上架待售？"}
    I -- "否（自留 / 種犬貓）" --> I1["維持在養 ACTIVE<br/>持續健康管理"]
    I1 --> Z2(["結束：日常照護循環"])
    I -- "是" --> J["狀態：待售 FOR_SALE<br/>設定售價與上架資訊"]
    J --> K["買家洽詢與看寵"]
    K --> L{"成交？"}
    L -- "否" --> K
    L -- "撤回上架" --> I1
    L -- "是" --> M["建立銷售訂單與付款"]
    M --> N["建立/綁定新飼主 Owner"]
    N --> O["交付準備：健康證明<br/>疫苗紀錄 / 登記過戶文件"]
    O --> P["狀態：已售出 SOLD<br/>完成交付與過戶"]
    P --> Q["發送交付通知與售後關懷<br/>站內 + Email"]
    Q --> Z3(["結束：售出結案"])
```

> 補充：SOLD 與 DECEASED 為終態，僅 ADMIN 以上角色可於誤操作時走「狀態更正」特別流程（需填寫原因並記 Audit Log）。

## 4. 官方登記申請流程

### 4.1 流程說明

- **觸發點**：租戶為寵物（RegistrationApplication）向主管機關申請官方登記，或買家要求交付時完成過戶登記。
- **關鍵決策**：文件是否齊全、晶片是否已植入並確認、主管機關審查結果（核准 / 補件 / 駁回）。
- **例外處理**：**補件為替代流程**——收到補件通知後於期限內補齊重送；逾期未補件視同撤件；駁回可申覆或重新申請。
- **狀態對應**：對應 [02_狀態機圖](02_狀態機圖.md) 的登記申請狀態機。

### 4.2 流程圖

```mermaid
flowchart TD
    A(["開始：發起登記申請"]) --> B["建立 RegistrationApplication<br/>關聯 Pet 與 Owner"]
    B --> C["準備文件<br/>身分證明 / 寵物照片 / 來源證明"]
    C --> D{"文件齊全？"}
    D -- "否" --> C
    D -- "是" --> E{"晶片已植入？"}
    E -- "否" --> E1["安排晶片植入<br/>由合作獸醫執行"]
    E1 --> E2["晶片植入確認<br/>掃描驗證晶片號碼"]
    E2 --> F
    E -- "是" --> F["送件至主管機關<br/>狀態：審查中"]
    F --> G{"審查結果？"}
    G -- "補件（替代流程）" --> H["收到補件通知<br/>站內 + Email 提醒承辦人"]
    H --> I{"期限內補齊？"}
    I -- "是" --> I1["補齊文件並重送"]
    I1 --> F
    I -- "逾期" --> I2["視同撤件<br/>狀態：已撤件"]
    I2 --> Z1(["結束：可重新申請"])
    G -- "駁回" --> J["記錄駁回原因"]
    J --> K{"申覆或重新申請？"}
    K -- "申覆" --> F
    K -- "重新申請" --> A
    K -- "放棄" --> Z2(["結束：申請關閉"])
    G -- "核准" --> L["取得核准函"]
    L --> M["發證：取得登記證<br/>登錄證號與發證日期"]
    M --> N["更新 Pet 登記狀態<br/>通知飼主與承辦人"]
    N --> Z3(["結束：登記完成"])
```

> 補充：每次狀態變更（送件、補件、核准、駁回、發證）均發送站內與 Email 通知，並記 Audit Log（含 before/after）。

## 5. 疫苗提醒流程

### 5.1 流程說明

- **觸發點**：系統每日排程（Cloudflare Queues / Cron Trigger）掃描各租戶的 Vaccination 排程，找出即將到期（預設前 14 / 7 / 1 天）與逾期的疫苗。
- **關鍵決策**：寵物目前狀態（已售出/已歿/已軟刪除不提醒）、通知偏好設定、是否已完成接種。
- **例外處理**：通知發送失敗進入重試佇列（最多 3 次）；飼主可設定延後提醒（snooze）；逾期未接種升級提醒 MANAGER。

### 5.2 流程圖

```mermaid
flowchart TD
    A(["開始：每日排程觸發"]) --> B["逐租戶掃描 Vaccination 排程<br/>強制帶 tenantId"]
    B --> C{"寵物狀態可提醒？<br/>排除 SOLD / DECEASED / 已軟刪除"}
    C -- "否" --> Z1(["跳過：不發送"])
    C -- "是" --> D{"到期判斷"}
    D -- "14 / 7 / 1 天內到期" --> E["產生到期提醒事件"]
    D -- "已逾期" --> F["產生逾期提醒事件<br/>同步升級通知 MANAGER"]
    D -- "未達提醒窗口" --> Z1
    E --> G["讀取租戶通知偏好"]
    F --> G
    G --> H["發送通知<br/>站內 + Email（LINE 推播規劃中）"]
    H --> I{"發送成功？"}
    I -- "否" --> J["進入重試佇列<br/>最多重試 3 次"]
    J --> K{"重試成功？"}
    K -- "否" --> L["標記發送失敗<br/>記錄於通知中心待人工處理"]
    K -- "是" --> M
    I -- "是" --> M["記錄通知歷程"]
    L --> Z2(["結束"])
    M --> N{"使用者操作？"}
    N -- "完成接種" --> O["更新 Vaccination 紀錄<br/>排定下一劑排程"]
    N -- "延後提醒 snooze" --> P["重設提醒時間"]
    N -- "無操作" --> Q["依窗口續發下一輪提醒"]
    O --> Z3(["結束：循環至下一劑"])
    P --> Z3
    Q --> Z3
```

> 補充：提醒窗口（14/7/1 天）與升級規則為租戶層級設定，預設值由系統提供，細節見 [26 通知中心](../26_通知中心/README.md)。

## 6. 訂閱升級 / 降級流程

### 6.1 流程說明

- **觸發點**：租戶 OWNER 或 ADMIN 於方案管理頁面發起方案變更。方案：Free NT$0、Starter NT$599、Pro NT$1,499、Enterprise NT$3,999 起（NT$/月）。
- **關鍵決策**：升級或降級、補差價付款是否成功、降級後配額是否超限。
- **計費規則（canon）**：**升級即時生效，按剩餘天數比例補差價；降級於下一計費週期生效**，期間功能維持原方案。
- **例外處理**：補差價付款失敗則升級不生效；降級時若現有資料超出目標方案配額（門市數、成員數、寵物數、儲存空間），須先處理超限項目或改為僅讀取。

### 6.2 流程圖

```mermaid
flowchart TD
    A(["開始：發起方案變更"]) --> B{"RBAC 檢查<br/>OWNER / ADMIN？"}
    B -- "否" --> Z0(["拒絕：403 無權限"])
    B -- "是" --> C{"變更方向？"}
    C -- "升級" --> D["試算補差價<br/>按本期剩餘天數比例"]
    D --> E["顯示金額並確認"]
    E --> F["執行付款"]
    F --> G{"付款成功？"}
    G -- "否" --> G1["升級不生效<br/>通知失敗原因，可重試"]
    G1 --> Z1(["結束：維持原方案"])
    G -- "是" --> H["方案即時生效<br/>解鎖新配額與功能"]
    H --> I["開立發票 / 收據<br/>寫入付款紀錄"]
    I --> J["發送變更通知<br/>站內 + Email"]
    J --> Z2(["結束：升級完成"])
    C -- "降級" --> K["檢查目標方案配額<br/>門市 / 成員 / 寵物 / 儲存"]
    K --> L{"超出目標配額？"}
    L -- "是" --> M["提示超限項目<br/>要求先行處理或確認降級後鎖定為唯讀"]
    M --> N{"確認繼續？"}
    N -- "否" --> Z1
    N -- "是" --> O
    L -- "否" --> O["排定降級：下一計費週期生效<br/>本期維持原方案功能"]
    O --> P["生效日前 7 天再次提醒"]
    P --> Q["週期日自動切換方案"]
    Q --> J2["發送變更通知<br/>站內 + Email"]
    J2 --> Z3(["結束：降級完成"])
```

> 補充：降級生效前可隨時取消排定；所有方案變更、試算金額與付款結果均記 Audit Log。付款細節見 [20 付款系統](../20_付款系統/README.md)、方案內容見 [19 會員訂閱](../19_會員訂閱/README.md)。

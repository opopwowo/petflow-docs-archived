# 時序圖

> 以 Mermaid sequenceDiagram 描述關鍵跨系統互動：登入驗證、下單（售出交易）與通知發送，明確標示各層職責與 Cloudflare 元件。

## 1. 文件目的

時序圖用於呈現**跨層、跨服務**的互動順序與職責邊界，補足流程圖無法表達的呼叫細節。所有互動遵循 Clean Architecture：Controller 極薄，流程編排在 Application 層 Service，商業規則在 Domain 層，資料存取透過 Repository 介面。

**共通約定：**

- 所有 API 走 `/api/v1/...`，預設需要驗證與授權（RBAC + Multi-Tenant）。
- 後端執行於 Cloudflare Workers，資料庫為 D1，快取為 KV，非同步任務走 Queues。
- 所有寫入操作記 Audit Log；錯誤回應採統一錯誤格式與正確 HTTP 狀態碼。

## 2. 登入時序圖

### 2.1 說明

- **觸發點**：使用者於登入頁輸入 Email 與密碼（或使用 SSO，Enterprise 方案）。
- **關鍵決策**：帳密是否正確、帳號是否啟用、租戶是否有效、是否啟用兩步驟驗證（2FA）。
- **例外處理**：連續失敗 5 次鎖定 15 分鐘；租戶停用回應 403；Token 過期走 Refresh 流程。

### 2.2 時序圖

```mermaid
sequenceDiagram
    autonumber
    actor U as 使用者
    participant FE as 前端（Pages）
    participant GW as API Gateway（Workers）
    participant AUTH as AuthService（Application 層）
    participant REPO as UserRepository（介面 / D1 實作）
    participant KV as KV（Session / 鎖定計數）
    participant AUD as AuditLogService

    U->>FE: 輸入 Email / 密碼
    FE->>GW: POST /api/v1/auth/login
    GW->>GW: Schema 驗證（Zod）
    GW->>AUTH: login(email, password)
    AUTH->>KV: 檢查鎖定狀態（失敗計數）
    alt 已鎖定
        KV-->>AUTH: 鎖定中（15 分鐘）
        AUTH-->>FE: 429 嘗試次數過多
    else 未鎖定
        AUTH->>REPO: findByEmail(email)
        REPO-->>AUTH: User（含 tenantId、角色、狀態）
        alt 帳密錯誤
            AUTH->>KV: 失敗計數 +1（滿 5 次鎖定）
            AUTH->>AUD: 記錄登入失敗
            AUTH-->>FE: 401 帳號或密碼錯誤
        else 租戶停用或帳號停用
            AUTH->>AUD: 記錄拒絕登入
            AUTH-->>FE: 403 租戶或帳號已停用
        else 驗證通過
            opt 啟用 2FA
                AUTH-->>FE: 要求 OTP
                U->>FE: 輸入 OTP
                FE->>AUTH: POST /api/v1/auth/verify-otp
            end
            AUTH->>KV: 建立 Session / 簽發 Access + Refresh Token
            AUTH->>AUD: 記錄登入成功（who / when / IP / 裝置）
            AUTH-->>FE: 200 Token + 使用者資料（角色、tenantId）
            FE-->>U: 進入儀表板
        end
    end
```

> 補充：Access Token 內含 `tenantId` 與角色宣告，後續每個請求由 Gateway 統一驗證並注入租戶上下文，禁止跨租戶存取。

## 3. 下單（寵物售出交易）時序圖

### 3.1 說明

- **觸發點**：門市人員（STAFF 以上）為狀態「待售 FOR_SALE」的寵物建立銷售訂單。
- **關鍵決策**：RBAC 權限、寵物狀態是否可售、付款是否成功。
- **例外處理**：寵物非 FOR_SALE 狀態回應 409；付款失敗訂單保留為待付款可重試；成交後以 Domain Event 觸發後續交付與通知。

### 3.2 時序圖

```mermaid
sequenceDiagram
    autonumber
    actor S as 門市人員（STAFF）
    participant FE as 前端（Pages）
    participant GW as API Gateway（Workers）
    participant OS as OrderService（Application 層）
    participant PET as Pet Aggregate（Domain 層）
    participant PREPO as PetRepository（D1）
    participant PAY as PaymentService / 金流閘道
    participant Q as Queues（Domain Event）
    participant AUD as AuditLogService

    S->>FE: 選擇寵物並建立訂單
    FE->>GW: POST /api/v1/orders
    GW->>GW: 驗證 Token + RBAC（orders:create）+ Schema 驗證
    GW->>OS: createOrder(petId, buyerInfo, tenantId)
    OS->>PREPO: findById(petId, tenantId)
    PREPO-->>OS: Pet
    OS->>PET: 檢查狀態可售？
    alt 非 FOR_SALE 狀態
        PET-->>OS: 不可售
        OS-->>FE: 409 寵物目前狀態不可售出
    else 可售
        OS->>OS: 建立訂單（含新飼主 Owner 建立/綁定）
        OS->>PAY: 建立付款單並請求收款
        PAY-->>OS: 付款結果（Webhook 驗簽 + 冪等）
        alt 付款失敗
            OS->>AUD: 記錄付款失敗
            OS-->>FE: 422 付款失敗，訂單保留可重試
        else 付款成功
            OS->>PET: 狀態轉移 FOR_SALE → SOLD
            OS->>PREPO: save(pet)（同一交易內落庫）
            OS->>AUD: 記錄售出（before/after、操作者、tenantId）
            OS->>Q: 發佈 PetSoldEvent
            OS-->>FE: 201 訂單成立
            FE-->>S: 顯示交付清單（健康證明 / 疫苗紀錄 / 過戶文件）
        end
    end
    Note over Q: 非同步：通知新飼主、產生交付文件、<br/>停止疫苗提醒排程、更新報表
```

> 補充：狀態轉移合法性由 Pet Aggregate（Domain 層）把關，Service 不重複實作規則；非法轉移一律回應 409。

## 4. 通知發送時序圖

### 4.1 說明

- **觸發點**：任何 Domain Event（如 `VaccinationDueEvent`、`PetSoldEvent`、`RegistrationStatusChangedEvent`、`SubscriptionChangedEvent`）進入通知中心。
- **關鍵決策**：租戶與使用者的通知偏好、管道可用性（站內 / Email / 推播，LINE 規劃中）、發送結果。
- **例外處理**：單一管道失敗不影響其他管道；失敗進入重試佇列（指數退避，最多 3 次）；最終失敗記錄於通知中心待人工處理。

### 4.2 時序圖

```mermaid
sequenceDiagram
    autonumber
    participant SRC as 事件來源（各模組 Service）
    participant Q as Queues（事件佇列）
    participant NC as NotificationService（通知中心）
    participant PREF as 偏好設定（KV / D1）
    participant INAPP as 站內通知（D1）
    participant MAIL as Email 服務
    participant PUSH as 推播（LINE 規劃中）
    participant AUD as AuditLogService

    SRC->>Q: 發佈 Domain Event（含 tenantId）
    Q->>NC: 消費事件（冪等處理，事件 ID 去重）
    NC->>PREF: 讀取租戶 + 收件人通知偏好
    PREF-->>NC: 啟用管道清單
    NC->>NC: 套用範本（繁體中文）產生通知內容

    par 站內通知
        NC->>INAPP: 寫入站內通知（未讀）
        INAPP-->>NC: 成功
    and Email
        NC->>MAIL: 發送 Email
        alt 發送失敗
            MAIL-->>NC: 失敗
            NC->>Q: 進入重試佇列（指數退避，最多 3 次）
            opt 重試仍失敗
                NC->>NC: 標記失敗，通知中心列管待人工處理
            end
        else 成功
            MAIL-->>NC: 成功
        end
    and 推播
        Note over PUSH: LINE 推播規劃中，<br/>目前僅預留管道介面
    end

    NC->>AUD: 記錄通知發送歷程（管道 / 結果 / 時間）
    NC-->>SRC: （非同步，無回呼）事件處理完成
```

> 補充：通知中心對每個事件以事件 ID 去重，確保 Queues 重送時不會重複通知；範本與管道細節見 [26 通知中心](../26_通知中心/README.md)。
